# 明账 iOS 设计系统

> 日期：2026-04-24
> 状态：讨论确认版
> 主题：定义明账 App 的 iOS 设计系统基础规范

---

## 1. 目标

这份文档回答一个问题：

`明账 App 应该以什么样的视觉、交互和组件系统呈现？`

它服务于后续 iOS App 产品设计、视觉稿、SwiftUI 组件实现和图表规范，不讨论数据库、API、自动分类或预算预测。

---

## 2. 已确认产品前提

`事实`

- 明账是个人财务管理 App，不是低龄化记账工具。
- 整体气质偏向独立 App 感，要求精致、专业、克制。
- 视觉参考方向是偏咨询风格的财务分析产品，而不是可爱、游戏化或营销化产品。
- 首页第一屏已确认采用 `收支结构上下对比`。
- 首页收支结构只统计收入类与消费/费用类支出。
- `资产类支出`、`负债类增记`、`负债类减记`、个人账户沉淀等进入资产负债或专项统计，不混入首页收支结构。
- 首页不引入 `账月阶段`，不维护 `待处理事项`，不突出 `上月延续`。
- `流水日记账` 是真源，资产、负债、收入、支出的最终计算都应从真源记录来。

`推断`

- 明账设计系统的核心任务不是让 App 显得热闹，而是让用户更快看清财务结构。
- 视觉层必须服务数据和账务语义，不能用装饰抢占数字、图表和记录本身。

## 3. 设计原则

### 3.1 明晰优先

`设计决策`

所有页面优先回答用户当前最关心的财务问题：

- 本月收入从哪里来
- 本月支出花到哪里去
- 当前资产和负债结构如何
- 某条账目如何参与计算
- 修改一条真源记录后结果如何变化

不为了视觉效果牺牲可读性。

### 3.2 专业克制

`设计决策`

- 不使用卡通插画。
- 不使用糖果色、强渐变或过度圆润的低龄化表达。
- 不做营销式大标题和情绪化文案。
- 不把财务 App 做成打卡、养成或游戏化产品。

### 3.3 数据优先

`设计决策`

- 图表、金额、分类结构和真源记录是界面主角。
- 视觉层级通过字重、间距、低饱和色和分割关系建立。
- 大数字可以存在，但不能把所有页面都做成单一 dashboard。

### 3.4 语义一致

`设计决策`

- 收入、支出、资产、负债有稳定的语义色和组件表达。
- 金额符号、收付类型、类型明细、收付手段不能被界面文案误导。
- 首页收支口径和资产负债口径必须清晰分离。

### 3.5 原生可信

`设计决策`

- 遵循 iOS 的导航、Tab、Sheet、搜索、输入、无障碍和安全区规则。
- 使用系统字体、SF Symbols 和 SwiftUI 原生交互。
- 自定义视觉服务品牌气质，但不破坏 iOS 用户熟悉的操作模型。

## 4. iOS 设计采用策略

### 4.1 平台基础

`设计决策`

- 以 iOS 26 及后续系统的设计能力作为前瞻方向。
- 保持对 iOS 原生模式的尊重：Tab bar 承接一级结构，NavigationStack 承接下钻，Sheet 承接聚焦任务。
- 设计时以最小 iPhone 宽度 `375pt` 作为基础约束。
- 交互热区不小于 `44pt × 44pt`。
- 页面左右基础边距为 `16pt`，高密度数据区可在局部使用 `12pt` 内边距。

### 4.2 Liquid Glass 策略

`设计决策`

Liquid Glass 用于界面层，不用于数据本体。

适合使用 Liquid Glass 的位置：

- Tab bar
- Navigation bar
- Toolbar
- Sheet 背景
- 右下角悬浮操作入口
- 悬浮筛选器
- 临时上下文菜单

不适合使用 Liquid Glass 的位置：

- 首页收支结构图表主体
- 真源账目列表行
- 大面积数据卡片
- 表格型明细区域
- 需要高对比阅读的金额和分类区域

`推断`

对明账而言，Liquid Glass 应该建立空间层级和现代 iOS 感，而不是制造装饰感。数据内容区应该保持稳定、实体、清晰。

### 4.3 可用性回退

`设计决策`

- iOS 26 以下使用系统 material 或普通实体背景回退。
- 当用户开启降低透明度或提高对比度时，玻璃层应退化为高对比实体背景。
- 任何关键金额、按钮和字段名都不能只依赖玻璃背景可读。

## 5. 信息架构

### 5.1 一级 Tab

`设计决策`

明账第一版采用 5 个一级 Tab：

1. `首页`
2. `账本`
3. `资产负债`
4. `统计`
5. `设置`

### 5.2 一级职责

`设计决策`

- `首页`：当前账月收支结构、摘要和高频入口。
- `账本`：真源流水、导入记录、补录记录、引擎生成记录的统一查看和编辑入口。
- `资产负债`：资产、负债、净资产、现金、投资、个人账户、待摊等存量结构。
- `统计`：跨月趋势、分类对比、明细分析和专项统计。
- `设置`：收付手段、分类体系、导入规则、显示偏好和数据管理。

### 5.3 全局入口

`设计决策`

右下角悬浮入口是全局高频操作入口，默认包括：

- `记一笔`
- `导入账单`
- `补录账务`
- `记还款`
- `记投资`
- `记摊销`

首页负责露出入口，具体操作进入对应表单或流程。

## 6. 颜色系统

### 6.1 基础色

`设计决策`

明账基础色偏石墨、冷灰和深墨蓝，避免单一蓝色或大面积紫蓝渐变。

| Token | 用途 | 浅色模式建议值 |
| --- | --- | --- |
| `color.bg.primary` | 页面背景 | `#F7F8FA` |
| `color.bg.secondary` | 次级背景 | `#EFF2F5` |
| `color.surface.primary` | 内容面板 | `#FFFFFF` |
| `color.surface.secondary` | 次级面板 | `#F3F5F8` |
| `color.text.primary` | 主文字 | `#111827` |
| `color.text.secondary` | 次级文字 | `#5B6472` |
| `color.text.tertiary` | 弱文字 | `#8A93A3` |
| `color.border.subtle` | 细分割线 | `#E3E7EE` |
| `color.brand.ink` | 品牌深色 | `#172033` |
| `color.brand.accent` | 品牌强调 | `#2F5F8F` |

`推断`

深色模式不直接反转固定 hex，而应使用 Asset Catalog 或语义色提供独立变体。

### 6.2 财务语义色

`设计决策`

| Token | 语义 | 建议值 |
| --- | --- | --- |
| `color.finance.income` | 收入 | `#2E7D5B` |
| `color.finance.expense` | 支出 | `#9B3A46` |
| `color.finance.asset` | 资产 | `#2F6F9F` |
| `color.finance.liability` | 负债 | `#A76B22` |
| `color.finance.neutral` | 中性资金流 | `#667085` |

规则：

- 收入色只表示收入来源，不直接表示“好”。
- 支出色只表示消费/费用支出，不直接表示“坏”。
- 资产色与收入色区分，避免把资产增加误读为收入。
- 负债色与支出色区分，避免把还款或负债变化误读为消费支出。
- 涨跌变化必须配合文字和符号，不只依赖颜色。

### 6.3 图表色板

`设计决策`

图表使用低饱和、多色系咨询风格色板：

| Token | 建议值 | 用途 |
| --- | --- | --- |
| `color.chart.navy` | `#243B53` | 主系列 |
| `color.chart.teal` | `#2F7E79` | 收入/稳健系列 |
| `color.chart.burgundy` | `#9B3A46` | 支出/费用系列 |
| `color.chart.ochre` | `#B9822E` | 负债/警示系列 |
| `color.chart.slateViolet` | `#5B5F97` | 投资/专项系列 |
| `color.chart.moss` | `#6F7F45` | 辅助系列 |
| `color.chart.steel` | `#6B8299` | 中性系列 |

同一图表内优先控制在 `3-6` 个颜色。

## 7. 字体与数字系统

### 7.1 字体

`设计决策`

- 使用系统字体 SF Pro。
- 不引入装饰字体。
- 所有正文、标题和按钮遵循 Dynamic Type。
- 中文界面优先保持左对齐，不使用两端对齐。

### 7.2 字号层级

`设计决策`

| 层级 | 用途 | 建议 |
| --- | --- | --- |
| `type.largeTitle` | 页面大标题 | 系统 Large Title |
| `type.title` | 页面标题 / 关键模块标题 | 20-24pt，Semibold |
| `type.sectionTitle` | 区块标题 | 16-17pt，Semibold |
| `type.body` | 列表正文 / 字段名 | 15-17pt，Regular |
| `type.caption` | 时间、备注、辅助说明 | 12-13pt |
| `type.amount.hero` | 首页摘要金额 | 28-32pt，Semibold |
| `type.amount.row` | 列表金额 | 15-17pt，Medium |

### 7.3 数字格式

`设计决策`

- 金额默认使用千分位：`12,860.35`。
- 人民币符号只在摘要、详情或跨币种场景显式显示：`¥12,860`。
- 列表金额右对齐，并使用等宽数字特性。
- 首页摘要金额可隐藏小数。
- 明细和计算详情保留小数。
- 负数使用 `-`，不使用括号。
- 零值显示为 `0`，不要用 `-` 代替真实零值。

SwiftUI 实现时，金额文本应使用等宽数字：

```swift
.monospacedDigit()
```

## 8. 布局、密度与栅格

### 8.1 基础栅格

`设计决策`

- 基础间距单位为 `4pt`。
- 常用间距为 `8 / 12 / 16 / 24 / 32pt`。
- 页面外边距默认 `16pt`。
- 列表行最小高度 `44pt`。
- 高频操作控件触控区不小于 `44pt`。

### 8.2 信息密度

`设计决策`

明账允许比普通生活 App 更高的信息密度，但不能像 Excel。

规则：

- 首页第一屏应完整容纳账月、摘要、收入结构和支出结构主体。
- 图表条目保持可读，不为了展示更多项压缩到不可读。
- 明细页可以更密，首页必须保留判断空间。
- 不使用卡片嵌套卡片。

### 8.3 圆角与边界

`设计决策`

- 内容面板圆角默认 `8pt`。
- 小标签圆角默认 `6pt`。
- 悬浮按钮和系统胶囊控件使用 capsule 或系统形状。
- 少用阴影，优先用背景层级、边界线和间距建立层级。

## 9. 材料、层级与运动

### 9.1 层级

`设计决策`

界面分三层：

1. `内容层`：金额、图表、列表、表单，使用实体背景。
2. `导航层`：Tab bar、Navigation bar、Toolbar，可使用系统玻璃或 material。
3. `临时操作层`：悬浮菜单、Sheet、Popover、上下文操作，可使用 Liquid Glass。

### 9.2 运动

`设计决策`

- 动效服务空间关系，不做炫技。
- 悬浮菜单展开可以使用轻量 morphing。
- 图表加载使用淡入或轻量过渡，不使用弹跳式娱乐动效。
- 修改记录后，相关金额变化可以短暂高亮，但不使用强烈闪烁。

---

## 10. 图表系统

### 10.1 图表原则

`设计决策`

- 图表优先用于解释结构和趋势，不用于装饰。
- 首页默认不用饼图。
- 财务结构优先用横向条形图。
- 趋势优先用折线图或柱状图。
- 资产负债可使用结构条、堆叠条或趋势线。

### 10.2 首页收支结构图

`设计决策`

首页第一屏使用上下两组横向结构条：

1. `收入结构`
   - 工作收入
   - 理财收入
   - 其他收入

2. `支出结构`
   - 生活必要开支
   - 文娱游购开支
   - 财务费用开支
   - 投资自身开支
   - 其他必要开支

每项展示：

- 分类名
- 金额
- 占比
- 横向占比条

### 10.3 其他图表

`设计决策`

| 场景 | 图表 |
| --- | --- |
| 月度收支趋势 | 折线图或柱状图 |
| 收入/支出跨月对比 | 分组柱状图 |
| 分类结构 | 横向条形图 |
| 资产负债结构 | 堆叠条或结构条 |
| 净资产变化 | 折线图 |
| 投资收益趋势 | 折线图 + RuleMark |
| Top N 明细 | 列表 + 条形占比 |

### 10.4 图表可访问性

`设计决策`

- 图表不能只靠颜色表达。
- 每个图表必须有文本摘要。
- 关键数据点需要 VoiceOver 可读标签。
- 小屏下优先保留标签和金额，弱化装饰性坐标轴。

## 11. 核心组件

### 11.1 导航与范围组件

`设计决策`

- `账月切换器`：显示当前账月，支持前后月切换和月选择。
- `Tab bar`：承接五个一级区域。
- `Search entry`：账本和统计页优先提供搜索入口，首页不抢主视觉。

### 11.2 首页组件

`设计决策`

- `收支摘要条`
  - 收入
  - 支出
  - 结余

- `收入结构模块`
  - 收入分类条
  - 金额
  - 占比

- `支出结构模块`
  - 支出分类条
  - 金额
  - 占比

- `悬浮操作菜单`
  - 记一笔
  - 导入账单
  - 补录账务
  - 记还款
  - 记投资
  - 记摊销

### 11.3 账本组件

`设计决策`

- `真源账目行`
  - 日期
  - 收付手段
  - 金额
  - 收付类型
  - 类型明细
  - 备注

- `记录来源标签`
  - 手动
  - 导入
  - 投资回填
  - 引擎

- `金额变化标签`
  - 表示较上期变化
  - 必须配合文字和符号

### 11.4 选择器组件

`设计决策`

- `收付手段选择器`
- `收付类型选择器`
- `类型明细选择器`
- `账月选择器`
- `日期时间选择器`
- `对象选择器`，用于贷款、待摊、投资标的等对象

## 12. 交互模式

### 12.1 下滑

`设计决策`

首页通过下滑承接更深信息：

1. 支出二级明细 Top 3 或 Top 5
2. 最近参与计算的真源账目
3. 收入明细
4. 资产负债摘要入口
5. 更完整的分类统计

### 12.2 下钻

`设计决策`

- 点击收入分类进入收入分类明细。
- 点击支出分类进入支出分类明细。
- 点击账目进入账目详情。
- 点击资产负债摘要进入资产负债页。

### 12.3 编辑

`设计决策`

- 账目详情页支持编辑。
- 列表左滑可提供编辑和删除，但删除必须确认。
- 修改真源记录后，相关计算结果应立即刷新，并用克制反馈提示。

### 12.4 Sheet

`设计决策`

以下场景优先使用 Sheet：

- 新增一笔
- 快速筛选
- 分类选择
- 收付手段选择
- 悬浮菜单展开后的二级动作

复杂流程可以在 Sheet 内使用 NavigationStack。

---

## 13. 表单与录入

### 13.1 基础字段顺序

`设计决策`

记账和补录表单默认字段顺序：

1. 金额
2. 日期时间
3. 收付手段
4. 收付类型
5. 类型明细
6. 备注
7. 账月

其中 `账月` 默认由当前上下文带出，不作为用户每次都要维护的字段。

### 13.2 场景化入口

`设计决策`

不同入口可以预填部分字段：

- `记一笔`：偏真实发生记录。
- `补录账务`：偏账单外事项、调整、确认。
- `记还款`：预设负债类减记路径。
- `记投资`：进入投资相关记录或投资台账回填。
- `记摊销`：进入待摊对象或摊销记录。

### 13.3 表单原则

`设计决策`

- 不为了“看起来简单”隐藏关键字段。
- 可以通过默认值、分组和预填降低输入成本。
- 字段缺失必须明确提示缺什么、为什么需要补。

---

## 14. 状态与反馈

### 14.1 基础状态

`设计决策`

设计系统必须覆盖以下状态：

- 空数据
- 加载中
- 计算中
- 导入中
- 导入失败
- 字段缺失
- 修改已保存
- 计算结果已更新
- 删除确认
- 无网络
- 本地数据异常

### 14.2 状态表达

`设计决策`

- 空状态用简洁文字和操作入口，不用大插画。
- 错误提示要说清楚影响范围。
- 删除、覆盖、批量修改属于高风险操作，必须确认。
- 计算中状态要克制，不遮挡用户阅读已存在数据。

---

## 15. 文案系统

### 15.1 语言气质

`设计决策`

文案应专业、简洁、可理解。

避免：

- 过度鼓励式文案
- 可爱化语气
- 营销话术
- 过重会计术语直接堆叠

### 15.2 术语使用

`设计决策`

- 首页使用用户可读的财务语言，例如 `收入结构`、`支出结构`、`结余`。
- 专业页保留真实科目语言，例如 `资产类支出`、`负债类减记`。
- `补录账务` 可以作为正式入口名。
- `真源记录` 可以作为内部或高级说明，但普通列表标题优先使用 `账目` 或 `流水日记账`。

## 16. 图标与品牌资产

- 分类图标可选，但不应成为主视觉。
- 不为每个消费分类设计可爱图标。

`待确认`

- 可探索深墨蓝底 + 几何线条 + 轻微玻璃层级。

## 17. 无障碍与适配

`设计决策`

- 支持 Dynamic Type。
- 支持深色模式。
- 支持提高对比度。
- 支持降低透明度。
- 不只用颜色表达财务语义。
- 所有关键图表提供 VoiceOver 文本摘要。
- 所有触控区域不小于 `44pt`。
- 小屏优先，不把 Pro Max 作为唯一设计基准。

## 19. 与已确认首页设计的关系

`事实`

本设计系统承接已确认的首页设计：

- 首页第一屏为 `收支结构上下对比`。
- 收入结构只包含 `工作收入 / 理财收入 / 其他收入`。
- 支出结构只包含 `生活必要开支 / 文娱游购开支 / 财务费用开支 / 投资自身开支 / 其他必要开支`。
- 资产负债另行统计，不混入首页收支结构。
- 高频动作通过悬浮入口承接。

`推断`

后续如果生成首页视觉稿，应优先检验：

- 收支结构是否一眼可读
- 图表是否比大数字更像主角
- 金额和占比是否清楚
- 悬浮入口是否可达但不抢主视觉
- Liquid Glass 是否只出现在导航和操作层

## 21. 当前结论

`事实`

- 明账 iOS 设计系统采用 `iOS 原生骨架 + 专业咨询风格视觉语言`。
- 首页优先表达收支结构，而不是账月阶段、待办或资产负债总览。
- 颜色、字体、布局、图表、组件、交互、表单和状态规范已经形成第一版约束。

`推断`

- 这套设计系统可以支撑下一步进入首页高保真视觉方向、核心组件设计或 SwiftUI 原型设计。
