# 新明账 iOS 产品设计到开发路线图

> 文档分类：Work Plan / 产品设计到开发路线图
> 状态：讨论沉淀中
> 日期：2026-05-08
> 范围：从已完成的 Excel 记账实践梳理与 iOS 低保真 PNG 原型，推进到 iOS MVP 产品开发。
> 暂不展开：TestFlight、公测、App Store 上架材料、审核、运营与发布节奏。

## 0. 当前阶段判断

`事实`

- 当前已经完成 Excel 记账实践梳理。
- 当前已经完成 iOS 关键页面低保真 PNG 原型，最新主要产物位于 `design/key-screen-low-fidelity/v5.1/`。
- 当前 MVP 权威入口是 `docs/prd/mvp-prd.md`。
- 当前核心规则已经分散沉淀在以下 Spec：
  - `docs/specs/record-and-month-spec.md`
  - `docs/specs/stock-flow-engine-spec.md`
  - `docs/specs/settings-and-semantics-spec.md`
  - `docs/specs/import-alipay-wechat-spec.md`
  - `docs/specs/investment-ledger-spec.md`
- 当前阶段不再继续扩展大而全产品想法，而是要把产品设计转成可开发、可验收、可对账的工程任务。

`推断`

接下来应从：

```text
产品设计稿
-> 开发规格
-> 数据与领域模型
-> 算法 / 引擎
-> iOS 纵向闭环
-> MVP 功能补齐
-> 对账与内测准备
```

逐步推进。

第一版所谓“后端”不应默认理解为云服务器。当前更合理的定义是：

```text
App 内领域服务层 / 本地数据服务 / 账务计算服务
```

除非后续明确要做多设备同步、账号系统或云备份，否则 v1.0 可以先按纯本地 App 设计。

## 1. 总目标

`推断`

新明账 iOS v1.0 的开发目标是：

```text
复刻当前 Excel 复式记账工作流
-> 让用户能在 iPhone 上完成日常入账、导入整理、资产负债查看、结果分析和必要设置
-> 让所有结果都能回到流水日记账解释
```

第一版不追求完整财务软件能力，不追求预算、预测、AI、多端协同或复杂看板。

### 1.1 v1.0 成功标准

`推断`

v1.0 达到可开发完成状态时，至少应满足：

- 用户可以完成手工记账、编辑、删除。
- 用户可以按账月查看流水。
- 用户可以导入并整理支付宝、微信账单。
- 用户可以查看首页摘要、资产负债、结果视图。
- 用户可以维护收付手段、收付类型、类型明细和语义标签。
- 用户可以维护基金投资明细，并让结果回填流水。
- 用户可以导出、备份、恢复数据。
- 所有结果视图都可以追溯到来源流水或投资明细。
- 关键计算结果能用 Excel 样例或人工样例对账。

### 1.2 当前阶段交付目标

`推断`

当前不是直接“写完整 App”，而是先交付一组开发输入：

- v1.0 功能范围清单。
- 页面与交互规格。
- 字段与状态规格。
- 数据模型设计。
- 领域服务边界。
- 算法 / 引擎规则清单。
- P0 纵向闭环实施计划。
- 关键验收场景矩阵。

## 2. 工作流分层

`推断`

产品设计到开发阶段拆成 6 条主线：

| 主线 | 职责 | 核心问题 |
| --- | --- | --- |
| 产品规格 | 把 PRD、Spec、PNG 转成开发可消费规格 | 做什么、不做什么、怎么验收 |
| iOS 前端 | 实现 SwiftUI 页面、导航、交互和状态 | 用户如何看、点、改、回溯 |
| 领域服务 / 后端 | 承接业务用例、协调数据和引擎 | 一个操作如何变成稳定账务结果 |
| 数据 | 定义本地模型、存储、迁移、备份恢复 | 数据如何保存、演进、导入导出 |
| 算法 / 引擎 | 实现复式语义解释和账月重算 | 流水如何推导资产、负债、收入、支出 |
| 关键功能 | 按优先级串起可用闭环 | 先交付哪条真实用户路径 |

## 3. 阶段路线图

### 3.1 阶段 A：开发输入冻结

目标：

```text
把“已经设计过”变成“开发可以开工”
```

#### 产品规格

要做：

- 冻结 v1.0 功能范围。
- 把 MVP PRD 中的 12 条关键验收场景拆成测试用例。
- 把五个一级入口和 v5.1 PNG 页面逐一编号。
- 明确每张页面的职责、入口、出口、主操作和不承接能力。
- 明确所有术语：账月、时间、流水、记录来源、收付手段、收付类型、类型明细、语义标签、骨架行、继承行。

产出物：

- `v1.0 功能清单`
- `页面清单与导航关系`
- `页面字段表`
- `页面状态表`
- `验收场景矩阵`
- `术语表`

交付标准：

- 每个功能都能追溯到 PRD 或 Spec。
- 每张 PNG 都能对应到一个页面或状态。
- 开发不需要猜“这个按钮点了去哪”。
- 任何未确认问题都标注为 `待确认`，不混入已确认范围。

#### iOS 前端

要做：

- 建立页面编号与 SwiftUI 页面模块的映射。
- 明确 Tab 结构：`首页 / 流水 / 资产负债 / 统计 / 设置`。
- 明确全局账月选择器在哪些页面出现。
- 明确新增、编辑、导入、筛选、搜索、详情页的进入方式。
- 补齐空状态、加载状态、错误状态、无数据状态。

产出物：

- `前端页面模块清单`
- `导航流转图`
- `页面状态说明`

交付标准：

- 可以按页面模块拆开发任务。
- 所有主路径都有明确返回关系。
- 首页、流水、资产负债、统计、设置不互相抢职责。

#### 领域服务 / 后端

要做：

- 定义 App 内领域服务边界。
- 明确每类用户操作对应哪个 Use Case。
- 明确哪些操作会触发引擎重算。
- 明确哪些结果只读展示，哪些结果可编辑。

产出物：

- `领域服务清单`
- `Use Case 清单`
- `读写边界说明`

交付标准：

- 前端不直接拼计算规则。
- 数据层不直接承接业务解释。
- 所有入账最终通过流水或投资明细形成真源。

#### 数据

要做：

- 定义最小数据对象。
- 明确字段类型、默认值、是否用户可见、是否可编辑。
- 明确初始化配置。
- 明确本地存储技术选型。

产出物：

- `数据模型草案`
- `字段字典`
- `初始化配置清单`
- `存储选型说明`

交付标准：

- 能覆盖手工记录、导入记录、投资回填、引擎骨架行。
- 不引入账月状态、结账状态、锁账状态。
- 可以支持后续 schema 迁移。

#### 算法 / 引擎

要做：

- 把 `stock-flow-engine-spec.md` 拆成可实现规则。
- 明确现金、负债、递延资产、投资资产四个账户家族的输入和输出。
- 明确 engine record 覆盖 key。
- 明确引擎触发、重算和跨月传播边界。

产出物：

- `引擎规则清单`
- `引擎测试样例`
- `重算边界说明`

交付标准：

- 每条规则都有输入、输出和示例。
- 重算不会重复追加骨架行。
- engine 记录参与计算，但不触发引擎。

### 3.2 阶段 B：工程基础与本地账本骨架

目标：

```text
先建可以持续开发的 App 骨架和本地账本基础
```

#### iOS 前端

要做：

- 创建 iOS 工程。
- 建立 SwiftUI Tab 壳。
- 建立基础导航结构。
- 建立最小组件：账月选择器、金额显示、记录行、空状态、筛选标签、来源标记。
- 接入本地假数据或 seed 数据，验证页面结构。

产出物：

- 可运行 iOS 工程。
- 五个一级入口空壳。
- 最小 UI 组件集合。

交付标准：

- 真机或模拟器可运行。
- 五个一级入口可切换。
- 页面结构与 v5.1 低保真方向一致。

#### 领域服务 / 后端

要做：

- 建立 Use Case 层。
- 建立 Repository 协议。
- 建立查询服务：按账月查流水、查首页摘要、查资产负债、查统计结果。
- 建立命令服务：新增记录、编辑记录、删除记录。

产出物：

- `LedgerService`
- `MonthQueryService`
- `DashboardQueryService`
- `RecordCommandService`
- `SettingsService`

交付标准：

- 前端通过服务层读写，不直接访问存储细节。
- 服务层可以替换底层存储实现。

#### 数据

要做：

- 选定本地存储方案。
- 建立第一版 schema。
- 建立默认配置 seed。
- 建立数据迁移版本号。

建议对象：

| 对象 | 用途 |
| --- | --- |
| `JournalRecord` | 流水日记账记录 |
| `AccountingMonth` | 账月索引，不包含结账状态 |
| `PaymentMethod` | 收付手段 |
| `PaymentType` | 收付类型 |
| `PaymentDetail` | 类型明细 |
| `SemanticTag` | 语义标签 |
| `ImportBatch` | 导入批次 |
| `InvestmentTransaction` | 基金投资交易事实 |
| `EngineRun` | 引擎重算记录或调试信息 |
| `BackupManifest` | 备份文件元信息 |

交付标准：

- 可以保存、读取、更新、删除流水。
- 可以初始化默认科目和收付手段。
- 可以区分 `manual / import / investment_feed / engine`。

#### 算法 / 引擎

要做：

- 先实现最小语义解释接口。
- 建立引擎输入输出模型。
- 建立单元测试样例。

产出物：

- `RecordSemanticInterpreter`
- `StockFlowEngine`
- `EngineResult`
- 引擎单元测试。

交付标准：

- 同一批输入得到确定输出。
- 可以在不启动 UI 的情况下跑通计算测试。

### 3.3 阶段 C：P0 纵向闭环

目标：

```text
先跑通一条真实账务路径，而不是横向铺满所有页面
```

推荐第一条闭环：

```text
新增一笔信用卡餐饮消费
-> 进入当前账月流水
-> 结果视图支出增加
-> 资产负债中对应信用卡负债增加
-> 修改或删除该记录后，全部结果同步变化
```

#### 前端任务

要做：

- 手工记账页：新增模式。
- 流水列表页：当前账月记录。
- 记录详情 / 编辑页。
- 首页摘要：当前账月收入、支出、结余、最近记录。
- 资产负债总览：资产、负债摘要。
- 统计总览：月度结果摘要。

交付标准：

- 用户能在 UI 中新增、查看、编辑、删除一条记录。
- 记录保存后立即出现在流水。
- 首页、资产负债、统计不是写死数据。

#### 领域服务 / 后端任务

要做：

- `CreateRecord`
- `UpdateRecord`
- `DeleteRecord`
- `RecalculateAffectedMonths`
- `QueryMonthLedger`
- `QueryDashboardSummary`
- `QueryBalanceSheetSummary`
- `QueryResultSummary`

交付标准：

- 每次真源记录变化后，触发相关账月重算。
- 删除记录后，相关引擎结果同步更新。
- 查询结果来自流水和引擎结果，不来自页面缓存。

#### 数据任务

要做：

- 完成 `JournalRecord` 的 P0 字段：
  - `账月`
  - `时间`
  - `收付手段`
  - `金额`
  - `收付类型`
  - `类型明细`
  - `备注`
  - `record_source`
  - `record_kind`
  - `carry_forward_role`
  - `engine_family`
  - `engine_key`
  - `object_key`
- 完成默认收付手段：
  - `电子钱包余额`
  - `广发卡`
  - `花呗`
  - `账务处理`
- 完成默认收付类型和类型明细的最小集合。

交付标准：

- 一条信用卡消费记录可以被完整保存。
- 一条 engine 骨架行可以被生成并覆盖。

#### 算法 / 引擎任务

要做：

- 实现信用支付形成负债。
- 实现现金支付影响现金类资产。
- 实现结果分类汇总。
- 实现同一账月内重算。
- 实现 engine record 覆盖。

交付标准：

- `广发卡 / 100 / 生活必要开支 / 伙食费` 能解释为：
  - 支出增加 100。
  - 广发卡负债增加 100。
  - 现金资产不变。
- 修改金额为 120 后，结果变为 120，不保留旧的 100。
- 删除记录后，相关结果归零或回到删除前状态。

### 3.4 阶段 D：P0 核心页面补齐

目标：

```text
让五个一级入口都从真实数据工作，而不是只完成一条路径
```

#### 首页

要做：

- 当前账月收入、支出、结余。
- 收支结构摘要。
- 最近记录。
- 快捷记账入口。
- 查找入口。

交付标准：

- 首页只做概览和入口。
- 首页不承接完整分析。
- 首页不展示资产负债总览和投资资产摘要。

#### 流水

要做：

- 账月列表。
- 账月工作区。
- 记录新增、编辑、删除。
- 筛选。
- 搜索。
- 来源标记。
- 继承行和骨架行轻量展示。

交付标准：

- 用户可以按账月查看全部记录。
- 用户可以识别手工、导入、投资回填、引擎记录。
- 用户可以进入单条记录详情。

#### 资产负债

要做：

- 资产总览。
- 负债总览。
- 资产详情。
- 负债详情。
- 从对象回溯来源流水。
- 从负债发起还款或利息补记。

交付标准：

- 信用支付、还款、负债成本都能体现在负债对象中。
- 资产负债入口不直接形成第二套账。
- 结果异常能回到流水解释。

#### 统计

要做：

- 月度结果总览。
- 收支结构。
- 资产负债变化。
- 投资结果入口。
- 分类详情。
- 来源回溯。

交付标准：

- 统计结果来自流水和引擎。
- 分类详情能追溯到来源流水。
- 不做高级图表看板。

#### 设置

要做：

- 记账配置。
- 数据管理。
- App 设置。

交付标准：

- 设置不承接日常记账主流程。
- 不做导入字段映射配置。
- 不做账月规则配置。
- 不做继承全局偏好配置。

### 3.5 阶段 E：P1 支付宝 / 微信导入整理

目标：

```text
把真实账单文件整理成可入账流水
```

#### 产品规格

要做：

- 明确支付宝导入字段映射。
- 明确微信导入字段映射。
- 明确导入预览页字段。
- 明确批量修改账月、收付手段、收付类型、类型明细的交互。
- 明确确认导入后的编辑边界。

`待确认`

- 支付宝、微信具体原始字段名需要用真实样例锁定。
- 是否保留原始交易号作为去重和追溯字段，需要样例确认。

#### 前端任务

要做：

- 账单导入入口。
- 来源选择：支付宝 / 微信。
- 文件选择。
- 候选记录预览。
- 单条编辑。
- 批量修改。
- 确认导入。
- 导入完成页。

交付标准：

- 用户可以看到导入前候选记录。
- 用户可以逐条或批量修改账月。
- 用户可以在确认后继续到流水中编辑。

#### 领域服务 / 后端任务

要做：

- `ImportBillFile`
- `ParseAlipayBill`
- `ParseWechatBill`
- `MapImportedRecordToJournalCandidate`
- `ConfirmImportBatch`
- `EditImportCandidate`
- `BatchUpdateImportCandidates`

交付标准：

- 确认导入前不写入正式流水。
- 确认导入后生成 `record_source = import` 的真源记录。
- 导入后触发引擎重算。

#### 数据任务

要做：

- `ImportBatch`
- `ImportCandidate`
- 原始字段保留策略。
- 导入确认状态。

交付标准：

- 导入批次可以被追溯。
- 候选记录和正式流水边界清楚。

### 3.6 阶段 F：P1 设置与语义配置

目标：

```text
让用户可以维护长期记账语义，但不把设置变成规则中心
```

#### 前端任务

要做：

- 收付手段列表。
- 收付手段新增、编辑、停用。
- 收付类型列表。
- 类型明细列表。
- 类型明细新增、编辑、停用。
- 语义标签编辑。

交付标准：

- 用户可以维护资产型、负债型、账务处理型收付手段。
- 用户可以维护收付类型和类型明细的归属关系。
- 停用对象不影响历史记录展示。

#### 领域服务 / 后端任务

要做：

- `CreatePaymentMethod`
- `UpdatePaymentMethod`
- `DisablePaymentMethod`
- `CreatePaymentDetail`
- `UpdatePaymentDetail`
- `DisablePaymentDetail`
- `UpdateSemanticTag`

交付标准：

- 配置变更影响后续记账。
- 历史记录保留原始语义可解释性。
- 不提供复杂自动分类规则中心。

#### 数据任务

要做：

- 默认配置 seed。
- 停用状态。
- 配置版本。
- 历史引用保护。

交付标准：

- 删除或停用配置不会破坏历史流水。
- 默认配置能支撑首次使用。

### 3.7 阶段 G：P1 基金投资明细账

目标：

```text
复刻当前基金投资明细账，并把结果回填流水
```

#### 前端任务

要做：

- 基金投资明细账入口。
- 标的列表。
- 交易记录新增、编辑、删除。
- 买入、卖出、收益、亏损结果展示。
- 月度回填结果展示。
- 从统计或资产负债跳入投资详情。

交付标准：

- 用户维护交易事实，而不是优先手工维护计算字段。
- 投资结果可以追溯到投资明细。
- 投资回填记录在流水中展示，但核心修改回到投资明细。

#### 领域服务 / 后端任务

要做：

- `CreateInvestmentTransaction`
- `UpdateInvestmentTransaction`
- `DeleteInvestmentTransaction`
- `CalculateInvestmentLedger`
- `GenerateInvestmentFeedRecords`
- `RecalculateInvestmentAffectedMonths`

交付标准：

- 修改投资交易后，相关账月回填流水被覆盖更新。
- 投资回填记录使用 `record_source = investment_feed`。

#### 数据任务

要做：

- `InvestmentTransaction`
- `InvestmentHolding`
- `InvestmentMonthlySummary`
- 投资回填记录关联。

交付标准：

- 买入、卖出、收益、亏损都有稳定数据表达。
- 非基金投资品类不进入 v1.0。

#### 算法 / 引擎任务

要做：

- 平均成本法。
- 买入入账金额计算。
- 卖出入账金额计算。
- 已实现收益 / 亏损计算。
- 月度回填流水规则。

交付标准：

- 买入：`入账金额 = 交易金额`。
- 卖出：`入账金额 = 交易份额 * 历史平均持有成本`。
- 卖出时使用入账金额回填流水，不能用交易金额替代。

### 3.8 阶段 H：P1 数据导出、备份、恢复

目标：

```text
保证个人财务数据可迁移、可备份、可恢复
```

#### 前端任务

要做：

- 数据导出入口。
- 创建备份入口。
- 从文件恢复入口。
- 恢复前确认页。
- 恢复结果页。

交付标准：

- 高风险恢复操作必须明确确认。
- 用户能理解恢复会覆盖或合并哪些数据。

#### 领域服务 / 后端任务

要做：

- `ExportData`
- `CreateBackup`
- `ValidateBackupFile`
- `RestoreBackup`

交付标准：

- 备份文件可以校验版本。
- 不支持的版本要给出明确错误。
- 恢复失败不能破坏当前数据。

#### 数据任务

要做：

- 备份文件格式。
- schema version。
- 数据完整性校验。
- 恢复策略。

交付标准：

- 重装 App 后可以通过备份恢复核心账本。
- 导出数据可以作为人工审计材料。

## 5. 最小纵向切片

```text

-> 首页显示支出增加
-> 资产负债显示信用卡负债增加
-> 统计显示生活必要开支增加
-> 从统计或资产负债回溯到流水
```

### 5.4 引擎范围

- 信用支付形成负债。
- 支出分类汇总。
- engine 记录覆盖。
- 当前账月重算。

### 5.5 验收样例

输入：

```text
账月：2026-04
时间：2026-04-15 12:30:00
收付手段：广发卡
金额：100
收付类型：生活必要开支
类型明细：伙食费
备注：午餐
record_source：manual
```

预期：

- 流水出现该记录。
- 统计中 `生活必要开支` 增加 100。
- 资产负债中 `广发卡` 负债增加 100。
- 首页支出增加 100。
- 现金类资产不减少。
- 修改金额为 120 后，上述结果同步变为 120。
- 删除记录后，上述影响消失。

## 6. 开发交付标准

### 6.1 产品规格交付标准

- 每个页面有职责说明。
- 每个页面有字段表。
- 每个页面有空状态、错误状态和主要交互状态。
- 每个功能有验收标准。
- 每个规则能追溯到 PRD、Spec、Excel 样例或明确标注为 `待确认`。

### 6.2 前端交付标准

- 页面与导航可在真机或模拟器运行。
- 页面数据来自本地数据和服务层。
- 不使用写死结果冒充计算结果。
- 所有可编辑表单都有保存、取消、校验和错误反馈。
- 所有回溯入口能到达流水或投资明细。

### 6.3 领域服务 / 后端交付标准

- 前端不直接实现账务计算。
- 所有写操作经过 Use Case。
- 真源记录变化后触发必要重算。
- 引擎记录不触发引擎。
- 读模型和写模型边界清晰。

### 6.4 数据交付标准

- 支持离线本地使用。
- 支持 schema version。
- 支持默认配置初始化。
- 支持备份、导出、恢复所需的完整数据表达。
- 删除、停用配置不破坏历史记录。

### 6.5 算法 / 引擎交付标准

- 计算结果可重复、可测试、可解释。
- 每个账户家族都有单元测试。
- 修改历史记录后，受影响账月可重算。
- engine key 能覆盖旧骨架行。
- 投资卖出使用平均成本法计算入账金额。

### 6.6 对账交付标准

- 至少用手工样例覆盖 12 条 PRD 验收场景。
- 至少用 Excel / CSV 样例覆盖导入和关键分类。
- 差异必须标注为：
  - `事实`：样例或文档已确认。
  - `推断`：规则合理但尚需更多样例。
  - `待确认`：需要用户确认或更多证据。
