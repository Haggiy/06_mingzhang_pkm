# 新明账关键页面低保真结构工作安排

> 文档分类：Work Plan / 下一阶段工作安排  
> 说明：本文只记录后续低保真页面结构工作，不作为 PRD 或 Spec 权威入口。

## 1. 目标

`事实`

- 当前已经完成 `首页 / 流水 / 资产负债 / 结果视图 / 设置` 五入口产品骨架设计。
- 当前已经明确 MVP 必做能力、明确不做能力和关键验收场景。
- 当前还没有进入页面视觉、iOS 实现、数据库或 API 设计。

`推断`

下一步应进入 `关键页面低保真结构设计`。

本阶段目标是回答：

- 每个入口打开后第一眼看到什么。
- 页面上有哪些信息区。
- 用户可以从哪里发起主要动作。
- 点击后流向哪里。
- 哪些能力明确不放在该页面。

本阶段不做：

- 高保真视觉。
- 颜色、字体、图标细节。
- SwiftUI 实现。
- 数据库模型。
- API 设计。
- 复杂交互动效。

## 4. 工作原则

`推断`

- 先结构，后视觉。
- 先主路径，后边界路径。
- 先用户能看懂的页面，再补内部机制。
- 入口不能互相抢职责。
- 所有入账最终回到 `流水`。
- `资产负债` 和 `结果视图` 只能解释，不成为第二套账。
- 首页不做待办中心。
- 设置不做复杂规则中心。

---

## 5. 工作步骤

### Step 0：复核 MVP 边界

动作：

- 读取 MVP 必须保留功能。
- 读取 MVP 明确不做功能。
- 读取关键验收场景。

要分析：

- 页面设计是否会引入已排除能力。
- 是否有入口职责冲突。
- 是否有必须覆盖但还没有页面承接的能力。

产出：

- 一份 `页面设计约束清单`。
- 明确本轮低保真设计不能越过的边界。

完成标准：

- 能清楚说出每个入口的 MVP 范围。
- 能清楚说出哪些能力不进入页面设计。

---

### Step 1：首页低保真结构

动作：

- 设计首页的信息层级。
- 设计首页下方浮动入口。
- 设计首页左滑进入配置和其他页面的入口关系。

要分析：

- 当前账月收支分析展示哪些最小指标。
- 最近记录展示几类信息。
- `记账` 和 `查找` 如何作为下方浮动入口出现。
- 首页如何避免变成结果视图或待办中心。

产出：

- 首页页面定位。
- 首页信息区列表。
- 首页主操作列表。
- 首页跳转关系。
- 首页低保真文字线框。

完成标准：

- 首页只包含 `当前账月收支分析 / 最近记录 / 快速入口`。
- 首页不展示资产负债总览。
- 首页不展示投资资产摘要。
- 首页不承接完整分析。

---

### Step 2：流水入口总结构

动作：

- 设计流水入口进入后的整体结构。
- 明确账月列表和账月工作区的关系。

要分析：

- 用户先选账月，还是默认进入当前账月。
- 账月列表展示哪些信息。
- 账月工作区如何成为核心记录行工作区。
- 如何避免账月列表变成账月状态管理。

产出：

- 流水入口结构。
- 账月列表低保真结构。
- 账月工作区入口关系。

完成标准：

- 账月只按 `YYYYMM` 组织。
- 不出现账月状态。
- 不出现结账、锁账、核对完成状态。

---

### Step 3：流水账月工作区

动作：

- 设计某个账月下的记录列表结构。
- 设计筛选、搜索、记录来源展示。

要分析：

- 记录行需要展示哪些字段。
- 时间如何展示到秒。
- 来源标记如何轻量展示。
- 继承行和骨架行如何被识别但不干扰阅读。
- 如何进入记录详情或编辑。

产出：

- 账月工作区信息区。
- 记录列表结构。
- 筛选和搜索入口。
- 记录行低保真结构。

完成标准：

- 用户能查看某账月全部记录。
- 用户能识别手工、导入、投资回填、引擎行。
- 用户能进入单条记录编辑。

---

### Step 4：记录编辑页

动作：

- 设计单条记录新增、查看、编辑结构。

要分析：

- 字段顺序如何安排。
- `时间 / 账月 / 收付手段 / 金额 / 收付类型 / 类型明细 / 备注` 如何编辑。
- 记录来源不同，编辑能力有什么差异。
- 继承行如何设置。
- 删除动作如何放置。

产出：

- 记录编辑页字段结构。
- 新增模式和编辑模式差异。
- 不同来源记录的编辑边界。
- 删除和继承操作位置说明。

完成标准：

- 手工行和导入行可编辑。
- 投资回填行提示回投资明细账编辑。
- 引擎行可查看但不作为普通手工行编辑。

---

### Step 5：支付宝 / 微信导入整理

动作：

- 设计支付宝导入整理流程。
- 设计微信导入整理流程。
- 设计导入后人工整理结构。

要分析：

- 导入入口放在哪里。
- 导入后如何展示待确认记录。
- 默认账月如何按实际时间预填。
- 如何逐条或批量修改账月。
- 如何补充分类、备注、收付手段。
- 如何确认进入流水。

产出：

- 支付宝导入整理低保真流程。
- 微信导入整理低保真流程。
- 导入记录整理列表结构。
- 批量改账月交互说明。

完成标准：

- 支付宝导入和微信导入分别能跑通。
- 跨月文件不会被强制全部归入当前工作账月。
- 导入后记录成为 `import` 来源的流水行。

---

### Step 6：资产负债入口

动作：

- 设计资产负债入口的三大区结构。
- 设计资产总览、负债总览、投资资产与投资明细账入口关系。

要分析：

- 资产总览展示哪些资产类别。
- 负债总览如何展示余额、利息、资金成本、还款。
- 信用支付实时形成的负债如何呈现。
- 投资资产如何进入基金投资明细账。
- 如何从对象回溯来源流水。

产出：

- 资产负债入口低保真结构。
- 资产总览结构。
- 负债总览结构。
- 投资资产入口结构。

完成标准：

- 资产负债不直接改余额。
- 还款和利息补记最终生成流水行。
- 投资交易在投资明细账管理，并回填流水。

---

### Step 7：基金投资明细账

动作：

- 设计基金投资明细账的输入和结果展示结构。

要分析：

- 输入字段如何组织：日期、标的名称、交易类别、交易金额、交易份额、备注。
- 计算字段如何展示：入账金额、持有成本、投资收益、持有份额等。
- 买入、卖出、收益、亏损如何回填流水。
- 如何从流水回到投资明细账。

产出：

- 基金投资明细账低保真结构。
- 投资交易编辑结构。
- 投资回填关系说明。

完成标准：

- 支持基金买入和卖出。
- 支持平均成本法结果展示。
- 支持投资结果回填流水日记账。

---

### Step 8：结果视图

动作：

- 设计结果视图四个职责区。

要分析：

- 月度结果总览展示哪些指标。
- 收支结构如何下钻到类型明细。
- 资产负债变化如何区别于资产负债入口。
- 投资结果如何只展示已实现结果。
- 如何回溯来源流水或投资明细账。

产出：

- 结果视图低保真结构。
- 月度结果总览结构。
- 收支结构结构。
- 资产负债变化结构。
- 投资结果结构。

完成标准：

- 不出现待处理任务。
- 不出现会计等式视图。
- 不出现预算预测。
- 所有结果可回溯来源。

---

### Step 9：设置

动作：

- 设计设置入口结构。
- 设计记账配置、数据管理、App 设置分区。

要分析：

- 收付手段如何维护资产型 / 负债型属性。
- 收付类型、类型明细如何默认使用当前 Excel 体系。
- 语义标签如何作为系统理解能力保留。
- 数据导出、备份、恢复如何放置。
- 哪些配置明确不出现。

产出：

- 设置入口低保真结构。
- 记账配置结构。
- 数据管理结构。
- App 设置结构。

完成标准：

- 不做导入字段映射。
- 不做账月规则配置。
- 不做继承全局偏好。
- 不做复杂自动化规则中心。

---

### Step 10：全局查找

动作：

- 设计首页下方浮动 `查找` 的最小路径。

要分析：

- 查找对象包括哪些：流水记录、投资明细、结果项、设置项。
- 查找结果如何分组。
- 点击结果如何跳转到来源页面。
- 查找是否允许直接编辑。

产出：

- 全局查找低保真结构。
- 查找结果分组。
- 查找跳转规则。

完成标准：

- 查找只定位，不直接修改数据。
- 查找结果能进入流水、投资明细账、结果视图或设置。

---

### Step 11：主路径串联检查

动作：

- 用关键验收场景串联所有页面。
- 检查页面之间是否存在断点。

要分析：

- 首页记账是否能落到流水。
- 支付宝 / 微信导入是否能进入流水。
- 信用支付是否能实时影响负债。
- 还款是否能减少现金和负债。
- 投资明细是否能回填流水。
- 结果视图是否能回溯来源。
- 设置是否只影响后续可选项和语义。

产出：

- 主路径流转表。
- 页面断点清单。
- 待确认问题清单。

完成标准：

- 12 条 MVP 验收场景都能被页面结构承接。
- 没有入口职责冲突。
- 没有引入 MVP 明确不做的功能。

---

## 6. 建议执行顺序

`推断`

建议按以下顺序推进：

1. 首页。
2. 流水入口总结构。
3. 流水账月工作区。
4. 记录编辑页。
5. 支付宝 / 微信导入整理。
6. 资产负债入口。
7. 基金投资明细账。
8. 结果视图。
9. 设置。
10. 全局查找。
11. 主路径串联检查。

原因：

- 首页决定启动方式。
- 流水是真源，必须优先定住。
- 导入整理是 MVP 必做，必须在流水后立刻处理。
- 资产负债和结果视图都依赖流水。
- 设置影响字段和语义，但不应主导页面结构。
- 查找和主路径检查适合最后统一收口。
