# 账月与流水记录 Spec

> 状态：权威 Spec  
> 范围：账月、时间、金额、记录来源、继承行和导入记录改账月规则

## 1. 账月

`事实`

- 当前 Excel 实践中，每个月对应一个 `流水日记账_YYYYMM`。
- 用户是在某个账月容器内完成当月记账、补录、结转和分析。
- 记录的实际发生时间和所属账月是两个不同字段。
- 当前实践没有强制关账、锁账或账月状态。

`推断`

MVP 中，`账月` 是强容器字段，格式采用：

```text
YYYY-MM
```

规则：

- 新增记录默认进入当前工作账月。
- 导入记录按实际时间预填账月。
- 用户可以逐条或批量修改导入记录的账月。
- 修改历史账月记录后，引擎按影响范围重算。
- 不引入 `已完成 / 待核对 / 处理中` 等账月状态。

## 2. 时间

`事实`

- 当前字段名常写作 `日期`，领域语义上应理解为时间戳。
- 时间支持到秒。
- 导入记录如果原始账单提供到秒，应保留到秒。
- 原始记录只有日期时，记为当天 `00:00:00`。
- 手工新增记录时，默认使用当前时间。

`推断`

MVP 用户可见字段可以命名为 `时间` 或继续沿用 `日期`，但内部语义按时间戳处理。

规则：

- 时间用于排序和追溯实际发生事实。
- 账月用于决定记录归属和引擎重算范围。
- 固定或继承性质的记录拿不到真实时间时，可使用约定时间占位。
- 后续知道更准确时间时，可以修改。

## 3. 流水记录字段

`事实`

当前最小用户可见字段为：

- `账月`
- `时间`
- `收付手段`
- `金额`
- `收付类型`
- `类型明细`
- `备注`

`推断`

MVP 内部还需要以下机制字段：

- `record_source`
- `record_kind`
- `carry_forward_role`
- `engine_family`
- `engine_key`
- `object_key`

这些内部字段不一定全部暴露给用户，但需要通过轻量视觉标记表达关键差异，例如来源、继承、骨架。

## 4. 金额语义

`事实`

- `金额` 是当前科目的带符号发生额。
- 正数表示该科目的正常发生。
- 负数表示该科目的反向发生、冲销、退款、回退或释放。
- 金额正负不直接等于收入或支出。
- 资产、负债、收入、支出的影响方向由 `收付手段 + 收付类型 / 类型明细 + 金额正负` 共同推导。

`推断`

示例：

```text
广发卡 / 100 / 生活必要开支 / 伙食费
```

表示伙食费正常发生 100，并由广发卡承担；产品实时解释为支出增加和广发卡负债增加。

```text
账务处理 / -120 / 资产类支出 / 长期待摊费用
```

表示长期待摊费用反向发生 120，即递延资产释放。

## 5. 记录来源

`推断`

MVP 的 `record_source`：

| 值 | 含义 | 编辑规则 |
| --- | --- | --- |
| `manual` | 用户手工新增或补录 | 可直接编辑 |
| `import` | 支付宝、微信导入并确认 | 可直接编辑 |
| `investment_feed` | 投资明细账回填 | 流水中展示，核心交易回投资明细修改 |
| `engine` | 引擎生成骨架行 | 流水中展示，修改来源或规则后重算 |

关键规则：

- `manual` 和 `import` 是用户可直接修正的真源记录。
- `investment_feed` 进入流水后是真源记录，但编辑入口在基金投资明细。
- `engine` 记录参与计算，不触发引擎。

## 6. 记录性质与继承用途

`事实`

当前 Excel 实践通过固定行跨月延续账务意图。

`推断`

MVP 使用两个字段表达：

```text
record_kind = normal / carry_forward
carry_forward_role = accounting_skeleton / recurring_record
```

含义：

- `normal`：普通记录。
- `carry_forward`：需要带到下月的继承行。
- `accounting_skeleton`：复式骨架，用于资产负债闭环。
- `recurring_record`：周期付款或重复记录，用于减少重复录入。

继承行不是任务提醒，也不是账月状态。它是流水记录的一种性质。

## 7. 继承字段规则

`推断`

继承行带到下月时，遵循：

```text
继承的是账务意图，不是简单复制上月实际发生结果。
```

字段处理：

| 字段 | 规则 |
| --- | --- |
| `账月` | 改为下月账月 |
| `时间` | 默认平移到下月同日同秒，月末行平移到下月月末 |
| `收付手段` | `账务处理` 可保留；等待真实支付的记录可留空 |
| `金额` | 固定金额可保留；需重新确认的金额可清空 |
| `收付类型` | 保留 |
| `类型明细` | 保留 |
| `备注` | MVP 先保留 |
| `record_kind` | 默认保留 |
| `carry_forward_role` | 保留 |
| `record_source` | 生成新记录时内部重建 |
| `engine_key` | 生成新记录时内部重建 |

## 8. 导入记录改账月

`事实`

当前账月是强容器，不等同于实际发生时间所在自然月。

`推断`

支付宝、微信导入整理时：

- 系统按实际时间自动预填账月。
- 用户可以逐条修改账月。
- 用户可以批量修改账月。
- 修改账月不改变原始时间。
- 确认导入后，记录进入对应账月流水。

边界：

- 不做复杂账月规则配置。
- 不因为导入文件跨月，就强制全部进入当前工作账月。
- 不把账月修改做成待处理任务中心。
