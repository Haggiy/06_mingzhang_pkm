# 支付宝 / 微信导入整理 Spec

> 状态：权威 Spec
> 范围：支付宝、微信账单导入，字段映射，账月预填与调整

## 1. 定位

`事实`

- 当前 Excel 实践的核心账单来源是支付宝和微信。
- 账单处理后的记录会进入对应账月的流水日记账。
- 用户确认导入字段映射不做用户配置，先钉死在系统内部。

`推断`

MVP 导入只做：

```text
支付宝账单导入和整理
微信账单导入和整理
```

导入目标不是保存原始底稿，而是把外部账单整理成可入账的流水记录。

## 2. 导入数据流

`推断`

最小流程：

```text
选择支付宝或微信账单文件
-> 系统读取原始字段
-> 系统按内置映射生成候选流水记录
-> 按实际时间预填账月
-> 用户补充或修正分类、备注、收付手段、账月
-> 用户确认
-> 生成 record_source = import 的流水记录
```

确认导入后，记录即成为可编辑真源记录。

## 3. 字段映射

`推断`

MVP 使用系统内置映射，把支付宝、微信账单整理为最小流水字段：

- `账月`
- `时间`
- `收付手段`
- `金额`
- `收付类型`
- `类型明细`
- `备注`
- `record_source = import`

规则：

- 导入原始时间保留到秒。
- 原始记录只有日期时，时间补为 `00:00:00`。
- 原始付款方式清洗为系统内的 `收付手段`。
- 原始交易说明、商户、商品说明等可合并进入 `备注` 或辅助整理字段。
- `收付类型` 和 `类型明细` 可以由用户补充或修正。
- MVP 不提供用户自定义字段映射配置。

`待确认 / 后续分析`

- 支付宝、微信具体原始字段名后续根据导入文件样例进一步锁定。
- 是否保留原始交易号作为去重和追溯字段，后续具体分析导入文件后确认。

## 4. 账月预填与修改

`事实`

账月是强容器，不必等同于自然月。

`推断`

导入整理区规则：

- 默认按实际时间预填账月。
- 支持逐条修改账月。
- 支持批量修改账月。
- 修改账月不修改实际时间。
- 确认导入后，记录进入被指定的账月流水。

边界：

- 不做复杂账月规则配置。
- 不因为一个导入文件跨月，就强制全部进入当前工作账月。
- 不引入待处理任务中心。

## 5. 整理能力

`推断`

MVP 导入整理区至少支持：

- 预览候选记录。
- 补充收付手段。
- 补充收付类型。
- 补充类型明细。
- 补充或修改备注。
- 逐条修改账月。
- 批量修改账月。
- 确认导入。
- 确认后在流水中继续编辑。

## 6. MVP 不做

`推断`

- 银行/信用卡导入及其他来源导入。
- 用户自定义导入字段映射。
- 复杂自动分类规则。
- 要求导入时一次性完成所有分类整理。
- 将原始账单文件作为长期唯一真源。
